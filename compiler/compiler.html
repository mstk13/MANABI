コンパイラとは → 最適化(ループ展開・インライン化 など)

「そもそもコンパイラとは?」から、性能を上げる最適化までを図で。

📖 ①コンパイラとは②3つの層③最適化 ④性能への効き方⑤ISAとの関係用語集

① そもそもコンパイラとは?(翻訳機)

🌐 たとえ:翻訳者
人が書く言葉(日本語)を、相手が分かる言葉(英語)に訳す翻訳者。コンパイラは、 人が書くソースコード(C, Rust など)を、CPUが分かる機械語(ISAの命令)に訳すソフト。

CPU は C 言語を直接は理解できない。理解できるのはISAの命令(機械語)だけ。 そこでコンパイラが「C → 機械語」へ変換する。下の流れ:

同じCコードでも、賢いコンパイラほど“速い機械語”に訳す。だから性能はコンパイラ次第でも変わる。

② コンパイラの3つの層

  • フロントエンド:コードを読み解いて木構造(AST)に。文法・型をチェック。
  • 最適化(ミドル):中間表現(IR)という“中立な形”に直し、速くする変形を何度もかける。ここが本題。
  • バックエンド:IR を実際のISA命令に翻訳(命令選択)し、32本のレジスタに変数を割り当て、命令を並べ替えて出力。

代表的な実装:GCC, LLVM/Clang(どちらも RISC-V バックエンドを持つ)。

③ 最適化(before / after を見る)

ボタンで「最適化前 → 後」を切り替え。各最適化が何を減らすかに注目。

(1) ループ展開(loop unrolling)

ループの中身を数回ぶん複製 → ループの戻り判定(分岐)とカウンタ更新の回数を減らす。分岐が減る=予測ミスのチャンスも減る。

for (i=0; i<100; i++) {   # 100回まわる=分岐100回
    a[i] = b[i] + c[i];
}
for (i=0; i<100; i+=4) {  # 25回=分岐1/4に
    a[i]   = b[i]   + c[i];
    a[i+1] = b[i+1] + c[i+1];
    a[i+2] = b[i+2] + c[i+2];
    a[i+3] = b[i+3] + c[i+3];   # 本体4つ=並列にしやすい
}
効果:分岐/カウンタ更新の回数↓、命令の並列性↑。代償:コードが大きくなる。

(2) インライン化(inlining)

小さな関数の呼び出しを、関数の中身そのものに置き換える → 呼び出し/戻り(分岐+スタック操作)が消え、さらに前後と一緒に最適化できる。

int sq(int x){ return x*x; }      # 関数
y = sq(a) + sq(b);   # 呼び出し2回(call/return=分岐)
y = (a*a) + (b*b);   # 中身を直接展開=call/return消滅
                     # さらに定数なら畳み込みも可能
効果:呼び出しの分岐/スタック↓、追加最適化が効く。代償:展開しすぎるとコード肥大。

(3) if変換(if-conversion → branchless)

小さな if を、分岐の無い「選択」に変換(ISAの条件命令=cmov/Zicond等を使う)。これが ISA 編の「分岐回避」とコンパイラが出会う所。

if (cond) x = a;     # 分岐(予測ミスの可能性)
else      x = b;
x = cond ? a : b;    # 条件選択命令へ(分岐なし)
   # RISC-V Zicond: czero.eqz/nez + or の3命令
効果:予測しにくい分岐のフラッシュ損を回避。代償:両方を計算するムダ(予測しやすい時は逆に損)。

他にも:定数畳み込み(2*3→6を事前計算)、デッドコード除去(使わない計算を消す)、 共通部分式の削除、命令スケジューリング(並べ替えてストール回避)など多数。

④ コンパイラは性能にどう効くか

🧮 ISA編と同じ鉄則:実行時間 = 命令数 × CPI × クロック周期。
コンパイラは実際に出す命令列を決めるので:
  • 命令数↓:インライン化・定数畳み込み・デッドコード除去・良いレジスタ割当
  • CPI↓:ループ展開/if変換で分岐(=予測ミス)を減らす、スケジューリングでストール回避
だから同じCコードでも -O0(最適化なし)と -O2 で実行時間が数倍変わることも。
同じプログラムの実行時間(イメージ):最適化レベルで変わる

⑤ ISA と コンパイラ の関係(役割分担)

🤝 ISA = 道具を用意する / コンパイラ = 道具を使う
  • ISA が Zicond(条件ゼロ)や M(乗除算), V(ベクタ) という命令を“提供”。
  • コンパイラが、ソースを見て 「ここは分岐回避にしよう」「ベクタ命令にまとめよう」と判断して、その命令を実際に出す。
前回の「分岐の頻度はソフト/コンパイラが制御」= まさにこれ(ループ展開・インライン・if変換)。 ISAに良い道具があっても、コンパイラが使わなければ宝の持ち腐れ。逆も然り。
