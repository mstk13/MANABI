# コンパイラとは → 最適化(ループ展開・インライン化 など)

図つき版(ブラウザ): **[compiler.html](https://mstk13.github.io/MANABI/compiler/compiler.html)**

## 1. そもそもコンパイラとは
**ソースコード(C, Rust など)を、CPUが理解できる機械語(ISAの命令)に翻訳するソフト**。
CPU は C を直接実行できない(分かるのはISA命令だけ)。流れ: ソース → **コンパイラ** → 機械語 → CPU実行。
同じCコードでも、賢いコンパイラほど“速い機械語”に訳す → 性能はコンパイラ次第でも変わる。

## 2. コンパイラの3つの層
- **フロントエンド**:字句/構文解析 → **AST(木)**、文法・型チェック。
- **最適化(ミドル)**:**IR(中間表現)** という中立な形に直し、速くする変形を何度もかける(本題)。
- **バックエンド**:IR を **ISA命令に翻訳(命令選択)**、変数を32レジスタに**割り当て**、命令を**並べ替え**て出力。

代表実装: **GCC**, **LLVM/Clang**(ともに RISC-V バックエンドあり)。

## 3. 主な最適化
- **ループ展開(unrolling)**:ループ本体を数回ぶん複製 → **分岐(戻り判定)とカウンタ更新の回数↓**、並列性↑。代償:コード増。
- **インライン化(inlining)**:小関数の呼び出しを中身に置換 → **call/return(分岐+スタック)消滅**、前後と一緒に最適化可能。代償:展開しすぎ=コード肥大。
- **if変換(if-conversion)**:小さな if を**分岐なしの選択**(cmov/Zicond)へ → 予測ミス回避。代償:両方計算するムダ(予測しやすい時は逆効果)。← ISA編の分岐回避と合流。
- 他:定数畳み込み、デッドコード除去、共通部分式削除、命令スケジューリング。

## 4. 性能への効き方(Iron Law)
> **実行時間 = 命令数 × CPI × クロック周期**

- **命令数↓**:インライン化・定数畳み込み・デッドコード除去・良いレジスタ割当
- **CPI↓**:ループ展開/if変換で**分岐(=予測ミス)を減らす**、スケジューリングでストール回避

同じコードでも `-O0`(最適化なし)と `-O2` で実行時間が数倍変わることも。

## 5. ISA と コンパイラ の関係
**ISA=道具を用意 / コンパイラ=道具を使う**。
ISA が Zicond / M / V を“提供”し、コンパイラが「ここは分岐回避」「ここはベクタ化」と判断して**実際にその命令を出す**。
「分岐の頻度はソフト/コンパイラが制御」= ループ展開・インライン・if変換のこと。良い道具も使われなければ宝の持ち腐れ。
