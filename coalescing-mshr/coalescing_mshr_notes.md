# コアレッシング / MSHR / ライトコンバイン・ライトバック まとめノート

「**バラバラのアクセスを1回に束ねて、遅いメモリを叩く回数を減らす**」3つの仕組みと、その IOMMU(PTW パイプライン)への応用。
図で触りたいときは → **インタラクティブ版**: [`coalescing_mshr_writeback.html`](coalescing_mshr_writeback.html)。

---

## 0. 大前提 — なぜ「束ねる」のか

すべての出発点は **メモリが遅い**こと。

- メモリ1アクセス ≈ **100 ns**。IOMMU は 400 MHz(1サイクル = 2.5 ns)なので、**1アクセス ≒ 40サイクル**待たされる。
- だから性能の鍵は「**メモリを叩く回数(トランザクション数)を減らす**」こと。
- 3つの技術は **まとめる対象が違うだけ**で、目的は同じ:

| 技術 | 束ねる対象 | ひとことで |
|---|---|---|
| コアレッシング | 隣接アドレスへの複数の**読み要求** | バラバラの読みを1バーストに |
| MSHR | 同一ブロックへの**飛行中のミス** | 後から来たミスを相乗りさせる |
| ライトコンバイン/バック | 隣接アドレスへの複数の**書き込み** | バッファに溜めて1回で吐く |

→ 「**読み・待ち・書き**」の3バリエーションとして覚えると整理しやすい。

---

## 1. コアレッシング(読みを束ねる)

**問題**: `0x100, 0x108, 0x110, 0x118` への小さな読みが別々に来ると、各々が独立トランザクション。アドレス/コマンドのオーバヘッドを4回払う。

**解決**: 同じバースト範囲(例 32B)に入る要求を**1本にマージ**し、1回のバーストで取る。

```
Req A,B,C,D  →[コアレッシング]→  1バースト転送 (0x100–0x11F 一括)
```

**効果**: メモリトランザクション 4→1。代表例が **AXI のバースト転送**。

> 💡 「連続している」という情報は、専用フラグではなく **トランザクションの長さ**(AXI の `AxLEN`/`AxSIZE`、PCIe の TLP Length)が表す。1本の大きなバースト = それ自体が「この範囲は連続」の宣言。

---

## 2. MSHR(飛行中の同一ミスを束ねる) ← パイプライン化の心臓部

**MSHR = Miss Status Holding Register**。

**問題**: ブロック X がミスしてメモリ取得を出した(=飛行中、~100 ns 待ち)。その間に**同じ X への別のミス**が来る。MSHR が無いと、同じ X を**何度も取りに行く**(冗長)。

**解決**: **MSHR という小さな表**に「X 待ち」を登録し、後続ミスを**相乗り(waiter 登録)**させる。

```
miss X(A) → MSHR に block=X 登録、フェッチ1本だけ発行
miss X(B) → 既に X が飛行中 → waiters に B を追加(フェッチ出さない)
miss X(C) → 同上、C を追加
fill 到着 → A,B,C に一斉配布、3ミス同時解決
```

**時間で見ると**: A が出した1本のフェッチ(~100 ns 飛行中)に、B・C が途中で相乗り → メモリアクセスは **1回**で3ミス解決。

> 💡 **なぜ「パイプライン化で必須」か**: 線速を出すには翻訳を**多数同時に飛ばす**必要がある。このとき飛行中のミスを追跡する仕組み(=MSHR)が無いと、同じブロックに何本もアクセスを出してメモリ帯域と PTW エンジンを食い潰す。「複数ミスを同時に処理する」こと自体が MSHR を要求する。

---

## 3. ライトコンバイン / ライトバック(書きを束ねる)

読みの裏返し。

- **ライトコンバイン**: 隣接アドレスへの複数の書き込みを、バッファ上で1つにまとめる。
- **ライトバック**: すぐメモリに書かず(←これが write-through)、バッファが**満杯/追い出し時にまとめて1回**書き戻す。

```
W1..W4 (0x200,204,208,20C) →[ライトバッファ]→ 1回のバースト書き戻し (0x200–0x20F)
```

> ⚠️ **トレードオフ**: 吐くまでメモリに反映されないので、順序・可視性の管理(**フェンス/フラッシュ**)が要る。
> IOMMU での効きどころ: **A/D ビット更新**や**ディスクリプタ書き戻し**の集約。

---

## 4. IOMMU 応用 — 3つが同時に効く場所

**舞台**: 800 GbE では連続 IOVA のページが **12.8 cyc ごと**に流れてきて、多くが IOTLB をミスする。

**鍵となる観察**: 近接ページのミスは**同じ葉ページテーブル(leaf PTE table)のブロックを欲しがる**。だから:

```
連続ページのミス (page0,1,2… 全部 leaf表 X)
   ↓ MSHR        : block X 飛行中、page0/1/2 を相乗り(冗長 walk 抑止)
   ↓ コアレッシング/バースト : leaf 表を1バーストで広く読む
   ↓ IOTLB を一括充填 → 後続ページはヒット化(walk 不要)
```

| 仕組み | IOMMU での対応 |
|---|---|
| コアレッシング | 連続 IOVA の葉 PTE を1バースト読みに統合(AXI バースト) |
| MSHR | 同じ葉表ブロック待ちのページミスを相乗り(冗長 walk を抑止) |
| ライトコンバイン/バック | A/D ビット更新・ディスクリプタ書き戻しをまとめて1回で吐く |

これが「パイプラインを止めずに 800 GbE を満たす」筋書き。

## 5. 落とし穴(鵜呑みにしない3点) ← ここまで理解できれば「使える」

図解は分かりやすい一方、そのまま信じると危ない所がある。設計で詰めるなら必須の注意。

### ① 「64 kB バースト」は物理連続が前提
- 葉表16枚は **VA 階層的には 1 GB 内に収まる**(L1 は 512 エントリ → 1 GB を指せる、`16 | 512`)。ここは OK。
- だが `16 | 512` が保証するのは **VA インデックスの並び**であって、**DRAM 上の物理レイアウトではない**。
- 16連続 L1 エントリの指す葉表は、それぞれ**任意の PPN** = 普通は**物理的に散る**。AXI/PCIe の1バーストは**物理連続領域**しか張れない。
- → 64 kB バーストを成立させるには **ページテーブル自体を物理連続に確保(pin + 連続割り当て)** する必要がある。「自動で成り立つ」のではなく**強制する前提**。

### ② IOTLB 容量 ≥ 先読み深さ
- 64 kB(= 8192 PTE)読んでも、IOTLB が 32 エントリなら**大半を捨てる**。
- → 「一括読み→全部 IOTLB へ」ではなく、**数エントリ先までを逐次補充するストリーミング先読み**に作り替えるのが現実解。

### ③ MSHR と深い先読みは役割が重なる
- 先読みが効くとミスが**疎**になる(関連シム結論: depth ≥ 16 で peakPTW = 1 = 並列 walk 1本で足りる)。
- ミスが疎 = 「同一ブロックへの同時ミス」が減る → **MSHR の旨味は条件付き**。
- MSHR が効くのは、**先読みが浅いとき** / **walk レイテンシ(Case C ≈ 486 cyc)がミス間隔を超えるとき**。

---

## 6. ひとことまとめ

3つとも「バラバラのアクセスを1回に束ねて、遅いメモリを叩く回数を減らす」仕組み。

- コアレッシング = **読み要求**を束ねる
- MSHR = **飛行中の同一ミス**を束ねる
- ライトコンバイン/バック = **書き込み**を束ねる

IOMMU では、連続ページのミスを **MSHR** で相乗りさせ、**バースト**で葉表を一括取得して **IOTLB を一括充填**する——これがパイプラインを止めずに 800 GbE を満たす鍵。ただし**物理連続性**と**IOTLB 容量**の2つが実装上の本当の制約になる。
