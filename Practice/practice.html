Practice — RISC-V で「命令を速くする」を実験する

コンパイラの最適化 と ISA 拡張 で、生成される命令がどう変わるかを実際にコンパイルして確かめる。全ツール OSS。

実験のながれ

cd Practice && make build → make compare で下の比較が出る(全部このマシンで実測)。

① 分岐回避(Zicond)— Clang

int sel(int cond,int a,int b){return cond?a:b;} を Clang でコンパイル。

rv64gc(分岐あり)

beqz a0, .L2   # 分岐!予測ミスで損
mv   a2, a1
mv   a0, a2
ret

rv64gc_zicond(分岐なし)

czero.eqz a1,a1,a0
czero.nez a0,a2,a0
or        a0,a0,a1
ret  # 分岐ゼロ=フラッシュ無し
命令数は同じ4でも、右は分岐が無い=パイプラインのフラッシュが起きない(特に予測しにくい分岐で速い)。

② ISA拡張で命令が激減(Zbb の cpop)— GCC

int pc(unsigned long x){return __builtin_popcountl(x);}

rv64gc 8 命令

addi sp,sp,-16
sd   ra,8(sp)
auipc ra,0x0
jalr ra        # ライブラリ呼び出し
ld   ra,8(sp)
sext.w a0,a0
addi sp,sp,16
ret

rv64gc_zbb 2 命令(-75%)

cpop a0,a0   # 1命令で数える
ret
ISA に cpop(Zbb)があると、ループ/呼び出しが 1命令に。命令数 8→2。

③ コンパイラ最適化 -O0 vs -O2 — GCC

配列総和 sum()(同じCコード):

最適化命令数(sum関数)
-O0(最適化なし)31 命令
-O212 命令(余計なロード/ストアを除去・ループを整理)

④ 同じコード・同じISA でもコンパイラで違う

sel(cond?a:b) を 両方 rv64gc_zicond でコンパイル:

コンパイラ結果
GCC 13.3分岐(beqz)… zicond を使わない
Clang 18czero(分岐なし)… zicond を活用
🤝 まとめ:速さは ① ISA(良い命令があるか)× ② コンパイラ(それを使うか)× ③ 最適化レベル の掛け算。 道具(ISA)があっても、コンパイラが使わなければ効かない。逆も然り。 → 理屈は [ISA編](https://mstk13.github.io/MANABI/riscv-isa/isa.html) と [コンパイラ編](https://mstk13.github.io/MANABI/compiler/compiler.html)。
