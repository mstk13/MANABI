# AXI Virtual Channel と IOMMU 設計メモ

> 本日の議論まとめ。AXI の ID / virtual channel の整理と、IOMMU 設計の経験則（順序・デッドロック・サイジング・AXI→CHI）を一枚に統合したもの。

---

## 1. Virtual Channel（仮想チャネル）とは

**1 本の物理リンクを、論理的に独立した複数チャネルとして共有する仕組み。**

- 物理配線は 1 組でも、その上に「互いに独立して進める複数の流れ」を作る。
- 各 VC が **独立したバッファ（キュー）とフロー制御** を持つのが本質。物理リンクは共有するが、バッファが別なので、片方が詰まってももう片方は進める。
- 狙いは **head-of-line blocking（先頭ブロッキング）の回避**。単一キューだと先頭が止まると後続の無関係なトランザクションも全部止まる。VC でキューを分ければ追い越せる。
- 出自は NoC / ルータ設計。デッドロック回避とスループット向上のために発展した概念。

---

## 2. AXI と ID

### 2.1 そもそも素の AXI に VC はない

- **AXI4 標準には virtual channel の概念は存在しない。**
- AXI が持つのは別の仕組み:**5 つの独立物理チャネル（AW/W/B/AR/R）** と **AxID による順序緩和**。
- VC は「AXI を相互接続 / NoC 上で運ぶときに、トランスポート層が提供する」もの。AXI プロトコル自体の機能ではなく、配送レイヤーの要件。

### 2.2 AxID = 順序保証の単位

| 関係 | 順序 |
|---|---|
| 同一 ID 同士 | 順序が保たれる（発行順にレスポンス） |
| 異なる ID 同士 | 順序保証なし（並べ替え・独立進行が可能） |

- プロトコルは「同一 ID = 順序維持」というルールだけを与える。**何に同じ ID を振るかは実装者の設計判断。**
- ID 幅は実装定義（設計パラメータ）。read（ARID）と write（AWID）の ID 空間は独立。レスポンスは対応 ID で返る（RID↔ARID, BID↔AWID）。

### 2.3 ID 割り当ての単位（実務パターン）

「順序保証を共有したいトランザクションの集合」が ID の単位。

1. **マスター（発行元）単位** — 相互接続が各マスターポートに識別ビットを付加し、レスポンスを正しい発行元へ返す。
2. **アウトスタンディング・スレッド単位** — 依存チェーン（walk の連続 PTE フェッチ）は同一 ID で順序維持、独立ストリーム（別デバイスの translation / 別 DMA）は別 ID で並列化。
3. **トラフィック種別単位** — レイテンシ重視とスループット重視を別 ID にして並べ替え・優先制御を許す。

**設計原則**
- 順序を守る必要がある最小集合 → 同一 ID にまとめる。
- それ以外で独立に進めたいもの → できるだけ別 ID に分ける。
- 細かく分けるほど並列性（アウトスタンディング能力）↑、ただしトラッキング資源（リオーダーバッファ等）のコスト↑。`MaxMstTrans` / `MaxSlvTrans` がこのトレードオフを表す。

---

## 3. AXI で VC（的な仕組み）が必要になる理由

1. **デッドロック回避** — AXI を NoC / クロスバーに載せると、トラフィッククラスが物理リンクを共有して相互依存でデッドロックし得る。クラスごとに独立バッファ（VC）を割り当て、依存ループを閉じさせない。
2. **QoS / トラフィック分離** — 低レイテンシ（walk の依存リード）と高スループット（大 prefetch / ペイロード）が同リンクを共有すると HoL blocking が起きる。VC で分ければ緊急トラフィックが大バーストを追い越せる。`AxQOS` は優先度ヒントに過ぎず物理分離はしないので、実効分離には VC が要る。
3. **帯域効率 / リンク共有** — 物理配線を増やさず、1 本の広いリンクを VC で多重化して複数ストリームを同時に流す。

### AXI 標準との関係（整理）

- **AXI4 標準**: VC なし。AxID（順序緩和）+ AxQOS（優先度ヒント）のみ。
- **実装側（相互接続 / NoC / ブリッジ）**: 下位トランスポートで VC を実装し、デッドロック回避・QoS・リンク共有を実現。
- **CHI 等の上位プロトコル**: VC / トラフィッククラスをより明示的に扱う。AXI を CHI ネットワークやベンダ NoC に載せると、その層が VC を提供。

## 4. IOMMU 設計の経験則

### 4.1 順序ルールとデッドロック

**① read が無駄に待たされる**
PCIe 順序ルールでは、後続 read(NP) は先行 write(P) を追い越せない（Producer-Consumer モデルを壊さないため）。だが IOMMU から見れば read の変換は本来すぐ着手できる（テーブルを引くだけ）のに、順序ルールで前の write 完了を待たされる。高性能 IOMMU は「順序を守らなくても結果が変わらないペア」を見極めて並べ替え／変換を先行させる。

**⑨ 変換のための walk 自身が詰まる循環（最も明快なデッドロック）**
IOMMU が変換するにはページテーブルを read する（walk = NP を発行）。だが変換対象が外からの write で、その write を捌くために walk が要る。なのに walk の read が詰まった同じ経路を通らないと進めない。
→「仕事を終わらせるために必要なリソースが、その仕事自身に塞がれている」状態。
→ 対策: **walk 用トラフィックに専用経路・専用バッファを確保**し、本流が詰まっても walk だけは前進できるようにする。

**⑩ キュー分離は必要条件であって十分条件ではない**
「P と NP を別キューに」は第一歩としては正しいが、NP キューが溢れて詰まれば止まる。read を出したいのに Completion が返らず未完了数が上限に張り付き、その Completion は別の詰まった経路を待つ…という循環。
→ 本当に安全にするには **各経路が他経路の完了に依存せず独立に前進できる** ことまで保証する。

### 4.2 変換のサイジングと性能

| # | 項目 | 要点 |
|---|---|---|
| ② | IOTLB 容量 | プラットフォームの物理メモリ量から逆算。ページサイズ・目標ヒット率からエントリ数とキャッシュ容量を見積もる。勘でやらない。 |
| ③ | walker 本数 | 相場は **10〜256 本**。walk は数十〜数百サイクルかかるので 1 桁台はボトルネック、増やしすぎても面積・電力の無駄で頭打ち。まずこのレンジで当たりをつける。 |
| ④ | ランダム IOVA | 完全ランダムだと IOTLB がほぼ毎回ミスし毎回 walk。キャッシュは局所性があって初めて効く。「ランダムでも高スループット」は筋が悪い要求。現実の連続バッファ DMA には局所性があり、それを前提に設計。 |
| ⑤ | IO ブリッジ結合 | 単体配置か IO ブリッジと密結合か。L2 共有すると効率↑だがコヒーレンシ・アービトレーション調停で複雑さ↑。 |

### 4.3 インターコネクトの世代交代（AXI → CHI）

前提: AXI も CHI も Arm 系オンチップ接続。AXI は古典的汎用バス、CHI(Coherent Hub Interface) はメニーコアのキャッシュコヒーレンシ前提の新しい設計。

- **⑥ AXI は先細り、CHI へ?** — コア数増大で AXI が手狭になり CHI へ移行する潮流（観測・トレンド確認）。
- **⑦ 同一 ID 順序固定がメニーコアの限界** — 同一 ID は発行順処理を保証、順不同で返せるのは異なる ID 間のみ。大量アウトスタンディングを捌くにはこの制約がボトルネック。順序の自由度を上げるには ID を大量に使い分ける必要があるが、ID 幅・トラッキングに限界 → AXI でコアを増やし続けるのは無理 → CHI へ。
- **⑧ AXI と virtual channel** — 物理 1 本でも論理的に複数チャネルへ。「変換用」と「データ用」を別 VC に分ければ ⑨ の「walk が本流に詰まらされる」を回避できる。AXI でどう実現するか（あるいは CHI の方が素直）という論点。

## 5. 全体ストーリー（一本の論理として）

1. IOMMU は「外からの write を捌くために自分が read(walk) を発行する」構造ゆえ、順序ルールとデッドロックが本質的につきまとう（①⑨）。
2. 素朴なキュー分離では不十分（⑩）。**walk 専用経路 / 仮想チャネルで「独立した前進」を保証** する必要がある（⑧）。
3. 性能側は **局所性を前提**（④）に、メモリ規模から IOTLB を（②）、レイテンシ隠蔽から walker 本数を（③）定量サイジングし、ブリッジ結合度で複雑さを調整（⑤）。
4. これらの要求（大量アウトスタンディング・トラフィック分離・順序の柔軟性）を **AXI の同一 ID 順序固定が満たしきれなくなりつつある**（⑦）。
5. → より素直に VC とコヒーレンシを扱える **CHI への移行** が視野に入る（⑥）。

**結論:** IOMMU の順序・デッドロック問題を性能を殺さず解くには「独立前進の保証」が要る。その器として AXI は限界が見え、CHI が次の選択肢になる。
