# ECCを実際に作るときの落とし穴 まとめノート

ロケット向けに ECC（メモリのビット化けを直す仕組み）の仕様を2つ作った
（**①制御CPU用** と **②AI異常検知アクセラレータ用**）。そのレビューで出た指摘を、
専門用語をかみ砕いて「**何が問題で・なぜ・どう直すか**」で整理する。

図で見たいときは → **インタラクティブ版**: [`ecc_design.html`](ecc_design.html)

---

## 0. 前提となる用語（ざっくり）

| 用語 | ひとことで |
|---|---|
| ECC | メモリのビット化けを検出・訂正する仕組み（パリティの強化版） |
| ビット化け(SEU) | 放射線などで 0↔1 が勝手に反転する現象（宇宙で多い） |
| SECDED | 1ビット化けなら直す／2ビット化けなら気づく（直せない）方式 |
| MBU(多重化け) | 1発で隣り合う複数ビットが同時に化ける |
| レジスタファイル(RF) | CPUが毎サイクル使う超高速の小メモリ。RF = Register File。速さに最も効く |
| フリップフロップ(FF) | 1ビットを覚える最小の記憶回路。RFやCPU内部の状態はFFの集まり |
| 耐性FF(DICE等) | 放射線で化けにくい特殊なFF。同じビットを二重に持ち合い、1発でも復元 → レジスタをECCの代わりに守れる（DICE = Dual Interlocked storage CEll） |
| SRAM | チップ内の作業用メモリ（レジスタより大きく少し遅い） |
| TCM | CPU直結の高速メモリ（Tightly-Coupled Memory）。キャッシュと違い必ず一定時間で読める（決定論向き） |
| ロックステップ | 同じ計算を2つのCPUで並走させ、答えが違えば異常と気づく |
| インターリーブ | 隣り合うビットを物理的に離して置く配置（MBU対策） |
| ゴールデンコピー | 「正しい値の原本」を壊れにくい別の場所に持っておく |
| WCET | 最悪でも何サイクルで終わるかの保証（リアルタイム制御に必須） |
| 重み / 活性化 | AIの学習済みパラメータ / 計算途中のデータ |
| パリティ | ビット化けを検出だけできる最小の仕組み。全ビットの偶奇を1ビットで記録（直せはしない）。ECCの原型 |
| シンドローム | ECCが計算する「どこが化けたかを指す番号」。0=正常、0以外=化けた位置 |
| 不揮発メモリ | 電源を切っても消えないメモリ（フラッシュ等）。重みの原本の置き場所 |
| フラックス | 放射線がどれだけ飛んでくるか（単位時間・面積あたりの粒子数）＝化けやすさ |
| 断面積 | メモリセル1個が放射線に当たって化ける“的の大きさ”＝当たりやすさ。フラックス×断面積で頻度が出る |
| スクラビング | 背景で全メモリを定期的に 読む→ECCで直す→書き戻す。化けが溜まる前に掃除する |

## 1. 制御CPU用ECCへの指摘（毎回同じ速さで・止まらず動くのが命）

### ① レジスタファイル“全部”にECCは、かえって遅くする
- **問題**：レジスタファイルはCPUが毎サイクル触る一番速い場所。ECCのチェックを挟むと最高速度が落ち、全命令が遅くなる。
- **なぜ**：ECCは「読んだ値を点検して直す」ひと手間。小さく超高速な場所ではこの手間が相対的に重い。
- **どう直す**：適材適所。レジスタは放射線に強い特殊フリップフロップ or 軽い検出（パリティ）で守り、本格ECC（SECDED）は大きめメモリ（SRAM/TCM）に使う。

### ② 「2ビット化けはロックステップで乗り切る」はメモリには効かない
- **問題**：ロックステップは2CPUの答えを見比べる仕組み。両CPUが同じメモリを読むと、メモリ化けは両方が同じく食らい、答えが一致して気づけない。
- **どう直す**：役割分担。CPUの計算ミス＝ロックステップ、メモリ化け＝ECC＋原本（ゴールデンコピー）から読み直し／領域リセット。

### ③ ECCを「同じ1サイクルに詰め込む」と速度が落ちる
- **問題**：メモリ読み出し＋ECC計算を同一サイクルにするとクロックが上げられないことがある。
- **どう直す**：合成して測り、入らなければ1段に分ける。ただし制御は「いつも同じ時間」が大事なので段数は固定（誤りの有無で時間を変えない）。

## 2. AIアクセラレータ用ECCへの指摘（誤りに強いが「見逃し」は厳禁）

### ① 「AIは1ビットくらい平気」は“だいたい”の話
- **問題**：上位ビット（大きい桁）が化けると値が桁違いに変わり、異常判定が逆転しうる。最悪は「本当の異常の見逃し」。
- **どう直す**：活性化も最低限の保護を残す。軽くするなら上位ビットだけ厚く守る。

### ② 「壊れた重みを読み直す」── どこから？
- **問題**：重みが2ビット化けて直せないとき、正しい重みがどこにも無ければ復旧不能。
- **どう直す**：重みの原本（ゴールデンコピー）を壊れにくい不揮発メモリに置き、再ロード経路を明記。

### ③ 「小分け」と「インターリーブ」は二重投資かも
- **問題**：どちらも「1発で2ビット死ぬ」対策。短時間飛行なら原因はほぼMBU（隣接同時化け）だけ → インターリーブだけで足り、両方やると面積のムダ。
- **どう直す**：想定する多重化けを先に決め、必要分だけ（短時間＝インターリーブ優先で容量節約）。
- 補足：累算器（毎サイクル変わる）はECCより軽い検出/再計算で守り、対象から外す。

---

## 3. 両方に共通する“穴”

### ★① 「どれくらい化けるか」の数字が無い（最重要）
- **問題**：「宇宙だからSECDED」「短時間だからスクラブ不要」と言葉では言えるが、1飛行で何ビット化けるかの数字が無い → 足りる／足りないを検証できない。
- **どう直す**：`化けやすさ(フラックス) × 当たりやすさ(断面積) × ビット数 × 飛行時間 ≈ 化ける数` でラフに見積もる。これが全判断の背骨。

### ② 「正しい原本」をどこに置くか
- **問題**：直せないときの復旧元（原本）が曖昧。
- **どう直す**：ゴールデンコピーの置き場所と読み直し手順を仕様に書く。

### ③ 「書き込むとき」のパリティ作りが未記載
- **問題**：読むとき直す話が中心で、書くときに符号（パリティ）を作って保存する経路が無い。
- **どう直す**：エンコード（書き込み）経路とタイミングも仕様に含める。

## 4. まとめ（問題 → どう直す）

| 対象 | 問題 | どう直す |
|---|---|---|
| CPU | レジスタ全部にECC＝遅い | レジスタは耐性FF/パリティ、ECCは大きめメモリに |
| CPU | 共有メモリ化けはロックステップで気づけない | メモリ化けはECC＋原本から読み直し |
| CPU | ECC同一サイクルで速度低下 | 測って必要なら固定段数で1段に |
| AI | 「誤りに強い」は平均の話 | 上位ビットは守る／見逃し厳禁 |
| AI | 壊れた重みの原本が無い | 不揮発に原本を置き再ロード |
| AI | 分割＋インターリーブが二重 | 想定する多重化けに合わせ必要分だけ |
| 共通 | ★化ける量の数字が無い | フラックス×断面積×ビット×時間で見積もる |
| 共通 | 復旧の原本の置き場所が曖昧 | ゴールデンコピーと手順を明記 |
| 共通 | 書き込み時の符号生成が未記載 | エンコード経路も仕様に入れる |

> **いちばんの学び**：ECCは「方式を選ぶ」より、**適材適所（速い場所には軽い保護）**・**復旧の原本**・**“どれだけ化けるか”の数字** の3つで設計の良し悪しが決まる。
