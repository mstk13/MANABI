## 1. パイプラインとは(流れ作業)
1つの**命令**(CPUがやる1作業)を一気に処理せず、工程に分けて**重ねて**流す仕組み。
洗濯を「洗う→乾かす→畳む」に分け、「乾かしてる間に次を洗う」と全体が速く片付くのと同じ。

## 2. 用語
| 用語 | 意味 |
|---|---|
| 段(ステージ) | 流れ作業の1工程。定番5段 = IF(取得)→ID(解読)→EX(計算)→MEM(メモリ)→WB(書戻し) |
| クロック/サイクル | CPUの拍子。1拍=1サイクル。1秒の拍数=**周波数**(GHz) |
| サイクル時間 | 1拍の長さ。**短い=速い** |
| fmax | 出せる最高周波数 = 1 ÷(一番遅い段の時間) |
| クリティカルパス | 論理回路で**一番時間がかかる道**。サイクル時間を決める |
| レジスタ(FF) | 段の境目でデータを1拍止めて渡す部品。毎段**固定コスト(オーバヘッド)** |
| CPI | 1命令あたり平均サイクル数。**小さいほど良い**(理想1) |
| ハザード | 流れが乱れてムダな拍が入る事態 |
| 分岐予測ミス | 次の命令の予想を外し、先に流した命令を捨てて(フラッシュ)やり直すこと |

## 3. なぜ段に分けると速くできるか
論理回路は「電気が通る道」。一番長い道(クリティカルパス)でサイクル時間が決まる。
**段に分ける=長い道を途中で区切る** → 1区間が短い → 1拍を短くできる → 周波数↑。

> **サイクル時間 ≈ (全体の論理 L) ÷ 段数 N + 固定コスト R**

- 前半 `L/N` = 1段の論理(役立つ部分)。**仕事の総量Lは不変、N で“分けている”だけ**なので N↑で1段は小さくなる。
- 後半 `R` = レジスタの固定コスト(毎段つく、減らない)。

## 4. でも頭打ち(④)
N を増やすほど `L/N` が小さくなり、やがて **1段の仕事 ≈ R** に。すると
`サイクル時間 → R` で **fmax は 1/R で頭打ち**(それ以上は速くならない=収穫逓減)。
目安:1段あたり ~8〜15 FO4 より薄くしても無駄。

## 5. 段数を決める「3つの力」
| 力 | 内容 | 効果 |
|---|---|---|
| **① 下限** | 目標の速さ(周波数)を出すには最低これだけ段が要る | 速さ↑したい→段を増やす |
| **② 頭打ち** | レジスタ固定コストで、ある所から速くならない | fmax が飽和(上限=1/R) |
| **③ 上限/逆効果** | 分岐ミスで**ムダ(CPI)**増・**電力**増 | 性能がピークを越えると低下 |

**性能 = fmax ÷ CPI**。fmax は上がるが頭打ち、CPI は深いほど悪化 → **性能はある段数でピーク(=最適段数)**。
浅すぎ=遅い、深すぎ=ムダ・電力で損。**だから「やみくもに深く」はダメ。**

## 6. 分岐予測ミスのコスト(なぜ深いとムダが増えるか)
1. 分岐の「どっち?」は**奥の段(例 EX)で判定**されるまで確定しない。
2. それまで CPU は**予想した方の命令を先に流す**(投機)。
3. IF〜判定段の間にいる命令 = **投機中の命令**。**深い=投機中が多い**。
4. 外すと**それを全部捨てる(フラッシュ)** → 捨てる数 ≒ **ペナルティ ≒ パイプ深さ**。

> **ムダ(CPI増分) ≒ 分岐の頻度 × 予測を外す率 × ペナルティ(≒深さ)**

## 7. 誰が何を制御できるか
| 要素 | 主な担当 | 手段 |
|---|---|---|
| 分岐の**頻度** | **ソフト/コンパイラ** | ループ展開・インライン化・if変換(branchless) |
| (頻度を下げる道具) | ハード/ISA | 条件移動(cmov)・述語実行・RISC-V **Zicond** 等を“提供” |
| 外す**率** | **ハード(予測器)** | 分岐予測器・BTB・RAS(当たればムダ0) |
| **ペナルティ** | **ハード(深さ)** | 浅いパイプ/早い段で判定 |

→ 「頻度=ソフト、外す率=予測器、ペナルティ=深さ」。ISA は branchless の手段を渡せる。

## 8. 予測ミスの“無駄”は何があるか
- ① **スループット低下**(フラッシュで空き拍=処理が遅れる)
- ② **電力の無駄**(捨てる命令の分も回路が動く)
- ③ **キャッシュ/TLB 汚染**(wrong-path が不要データを載せ有用データを追い出す)
- ④ **セキュリティ(Spectre 系)**:投機の痕跡が漏れる経路に → 防衛/セキュアで致命的
- ⑤ **資源占有**(OoO):発行枠・LSU/物理レジスタ等を wrong-path が一時占有
- ⑥ **決定論/WCET 悪化**:実行時間がばらつく → 最悪時間が読めない(制御系で重大)
- ⑦ **設計・検証の複雑化**:squash・レジスタ復元・ストアバッファ巻き戻し等

## 9. ドメイン別の指針(このプロジェクト)
| 分野 | 方針 | 理由 |
|---|---|---|
| 宇宙/航空 制御(AMANOGAWA/SOUTEN) | **浅め(5段)・非投機寄り** | WCET/決定論が要件。深い投機は時間が読めず不利 |
| 防衛(SAKIMORI) | 投機を**制御/抑制** | Spectre 等のセキュリティリスク |
| 高性能/HPC(DENNO) | **深め + 強い予測器** | スループット最優先。外す率を下げてペナルティ増を相殺 |

## 10. 実機で確かめる
段数を変えた自作RTLを、**OpenROAD フローで fmax/電力/面積を実測**して最適段数を選べる
→ [SHISAKU/docs/physical_design_flow.md](https://github.com/mstk13/SHISAKU/blob/main/docs/physical_design_flow.md)(`run_flow.sh`)。
「段数だけ変えて回す→fmaxが飽和し電力が悪化する点」を見れば、3つの力が数字で確認できる。
