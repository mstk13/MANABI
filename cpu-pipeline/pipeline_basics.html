① パイプラインとは?(流れ作業)

🧺 たとえ:洗濯物
洗濯を「洗う→乾かす→畳む」の3工程に分けるとします。1人ぶんを全部終えてから次の人…だと遅い。 でも「Aさんを乾かしてる間に、Bさんを洗う」と工程を重ねると、全体がどんどん速く片付きます。これがパイプライン。

CPU も同じ。1つの命令(=CPUがやる1つの作業)を一気に処理せず、いくつかの工程に分けて重ねて流します。

② 段(ステージ)とクロック

段(ステージ)= 流れ作業の1工程。CPUの定番は5段: IF(命令を取る)→ID(解読)→EX(計算)→MEM(メモリ)→WB(書き戻し)。

クロック= CPUの“拍子”。1拍 = 1サイクル。1秒間の拍数が周波数(GHz)。 サイクル時間= 1拍の長さ。短いほど速い。1拍ごとに各命令が次の段へ1つ進みます。

下図:命令 I1〜I5 が段を流れる様子(横=サイクル, 色=段)。斜めに重なる=パイプライン。
IF ID EX MEM WB 　→ サイクル3以降は毎拍1命令が完成(重なってるから速い)

③ なぜ段に分けると“速くできる”のか

CPUの中身は論理回路(電気が通って計算する道)。電気が端から端まで通るのに時間がかかります。 その一番長い道をクリティカルパスと言い、サイクル時間はこの最長の道で決まります(間に合わないと壊れる)。

💡 段に分ける=長い道を途中で区切る。1区間が短くなる→1拍を短くできる→周波数(速さ)を上げられる。
ざっくり: サイクル時間 = (全体の論理) ÷ 段数 + すこしの固定コスト。段数を増やすと前半が小さくなる=速くなる。

④ でも“頭打ち”になる(レジスタの固定コスト)

段と段の境目にはレジスタ(フリップフロップ)という「データを1拍だけ止めて次へ渡す部品」が要ります。 これ自体に毎段固定の時間(オーバヘッド)がかかります。

だから段を増やしすぎると、1段の“役立つ計算”より“境目の固定コスト”の方が目立ってきて、もう速くならない(頭打ち)。下のバーで体感:

役立つ論理(段数↑で薄くなる) レジスタの固定コスト(変わらない)→ 比率が増える=頭打ち

⑤ “ムダ”の話(CPI と 分岐ミス)

CPI= 1命令あたり平均何サイクルかかるか。理想は1(毎拍1命令完成)。 でも実際は“流れが乱れる”=ハザードでムダな拍が入り、CPIは1より大きくなります(大きい=遅い)。

代表が分岐予測ミス。CPUは「次どの命令を実行?」を予想して先に流しますが、 分岐(if文など)で外すと、先に流した命令を全部捨ててやり直し。下図の×が捨てる命令です。

🔴深い(段数が多い)ほど、捨てる命令が増える → ムダ(CPI)が増える。これが「深くしすぎると損」の正体。

⑥ まとめ:段数を決める「3つの力」

ここまでの用語で、3つの力がそのまま読めます。スライダーを動かすと曲線が変わります。

速さ(クロック上限 fmax) ムダ(CPI) 実力(=速さ÷ムダ:本当の処理能力) 電力
力意味(やさしく)図の効果
①下限欲しい速さ(周波数)を出すには、最低これだけ段がいる速さ(青)を上げたい→段を増やす
②頭打ちレジスタの固定コストで、ある所から速くならない青がだんだん平らに
③上限/逆効果分岐ミスでムダ(赤)が増え、電力(紫)も増える緑(実力)がピークを越えると下がる
🎯 結論:「実力(緑)が一番高くなる段数」=最適。それより浅い=遅い、深い=ムダと電力で損。 だから「やみくもに深く」はダメ。
※ 宇宙/航空の制御チップは「動作時間がきっちり読めること(=予測=深い投機を避ける)」が大事なので、あえて浅め(5段くらい)で固定するのが正解、という話にもつながります。
