3
Llama モデルの理解
decoder 1層 = 7重みGEMM + 2 Attnコア。混ぜる/考える
▼

やったLlama decoder 1層を分解:重みGEMM 7本(Q/K/V/O射影・FFN gate/up/down)+ Attnコア2本(QKᵀ, ·V)。演算の8〜9割がGEMM。

やったモデルの動きを2本柱で理解:Attention=通信(トークンを横に混ぜる) / FFN=計算(各トークンを個別に深める)。

コヒーレンスを実装(概念→RTL)
MSI/MOESI Python → RTL HN/RN → 本物 OpenLLC 読解
▼

やった状態遷移を手で: MSI(coherence_min.py)→ MOESI(moesi.py, E=バス削減/O=書戻し削減)。トレース+assertで「古い値を読まない/Mは1つ」を検証。

やったRTL版 CHIノード:rn_coh_fsm/rn_node/hn_node/chi_mini を SystemVerilog で実装、cocotb で古典シナリオ・ping-pong 全PASS。

やった本物 OpenLLC(Chisel)を読解:self+client の2ディレクトリ・6段MainPipe・バンク分割。自作 hn_node との対応表を作成。

図: 状態機械(クリック遷移) code: coherence_min(Python+RTL)
6
CHI を 0 から実装(はしご)
R1 リンク層(credit)→ R2 トランザクション(TxnID)
▼

やったクリーンな再スタート。R1 = リンク層:flit を L-credit 流量制御で流す(credit送信+FIFO受信)。トークン保存則でオーバフロー無し、cocotbで40 flit順序通り。

やったR2 = 1トランザクション:RN→SN の ReadNoSnp→CompData を REQ/DAT 2チャネルで。opcode + TxnID 照合で要求と応答を結ぶ。cocotb PASS。

なぜ「リンク層は固定、チャネルとopcodeを足す」= CHI実装の進め方。R3(HN+SNPでコヒーレンス)へ続く。

code: chi(R1/R2 RTL+cocotb) はしごロードマップ
7
Tenstorrent 実スタックで実験(tt-sw)
ハード無し ttsim で ttnn op / LLK / Llama FFN
▼

TT-NN層ハード無し(ttsim)で 11 op 全PASS(add/sub/mul/matmul/exp/relu/gelu/sqrt/reciprocal/silu)。

op網羅(C)各opをプロセス隔離し 11/12。rms_norm 動作・layer_norm はttsimクラッシュ(=Llamaは RMSNorm なので好都合)。softmax/reduce/transpose も動作。

LLK層(B)ソース版 pytest --run-simulator:datacopy F32→F32 全PASS、F32→F16+dest_acc は ttsim 未実装。ハード最寄り層を実地確認。

Llama FFN(A)SwiGLU FFN down(SiLU(x@Wg)⊙(x@Wu)) を ttnn で連結し ttsim 実行、各段が解析 golden と一致。

図: TTスタック(再生) code: 実験3本
