# 物理設計(Physical Design)— 論理合成のあとの工程

> RTL → **論理合成** → ★ここから先★ → GDS。
> 合成が出すのは「どのセルを使い・どう繋ぐか」だけのネットリスト(位置も配線もクロックも無い=**地図のない部品表**)。
> 物理設計(PD)は、その部品を**シリコン上の座標に落とし込み、現実の遅延を一つずつ確定させていく**作業。
> 検証は実際の OpenROAD-flow-scripts(ORFS)で gcd / sky130hd を流した結果を使う。

---

## 0. 全体像:PDは「理想を1つずつ現実に置き換える」

合成直後のタイミングは「理想」だらけ — セルは点に重なっていて配線長ゼロ、クロックは全FFに同時到達する前提。
PDの各段は、この理想を1つずつ現実に差し替える。差し替えるたびに遅延が増えるので、**段が進むほどタイミングは普通悪くなる**。
だから各段の後で必ず **STA(タイミング解析)を回し直す**。これがサインオフまでの基本リズム。

| 段 | ORFS | 工程 | 新しく「現実」になるもの |
|---|---|---|---|
| 1 | `1_synth` | 論理合成 | セルと接続(全部理想) |
| 2 | `2_floorplan` | **フロアプラン** | ダイ/コア外形・I/O位置・電源網(PDN)・タップセル。クロックはまだ理想 |
| 3 | `3_place` | **配置(Placement)** | 全セルに実座標 → 配線長が見え始める |
| 4 | `4_cts` | **クロックツリー合成(CTS)** | 実クロックツリー(skew・挿入遅延が発生) |
| 5 | `5_route` | **配線(Routing)** | 実配線のRC(抵抗・容量)が乗る |
| 6 | `6_final` | **仕上げ・署名(Finish/Signoff)** | フィラー充填・寄生抽出・最終STA・GDS出力 |

## 1. gcd で実測したタイミングの推移(この表が核心)

ORFS の各段レポート(`reports/sky130hd/gcd/base/*.rpt`)から拾った実数字。
**要求クロック period = 1.10 ns(≒909MHz)** に対する worst slack(WNS)と、回路が実際に出せる fmax。

| 段 | WNS (ns) | fmax (MHz) | TNS (ns) | コメント |
|---|---|---|---|---|
| 2 floorplan | **-2.04** | 318.5 | -78.6 | クロックはまだ「理想」。電力もクロック網=0 |
| 3 global place | -2.18 | 304.5 | -91.2 | セルに座標が付き配線長が乗る |
| 3 detailed place | -2.18 | 304.5 | -93.3 | 行に整列・重なり解消(legalize) |
| 4 CTS | -2.21 | 301.8 | -94.3 | 実クロックツリー挿入。skew=0.01ns |
| 5 global route | **-2.42** | 283.9 | -102.2 | 実配線RCが初めて正確に入る → **最悪点** |
| 6 finish(署名) | -2.26 | 297.6 | -95.5 | 最終最適化で少し回復。**DRC違反 0** |

**読み取れること:**
1. **段が進むほど slack が悪化** — 座標→配線長→クロックツリー→配線RC、と現実が増えるたび遅延が増える。
   配線後(-2.42)が最悪なのは、ワイヤの抵抗・容量が初めて正確に入るから。
2. **各段の後で STA を回し直している** — だからこの履歴が取れる。「合成で1回見て終わり」ではない。
3. **このgcdはタイミング未達(教材的)** — 要求 1.10ns に対し限界は 3.36ns(fmax≒298MHz)。
   GDSは出るが署名STAは赤(setup違反60本)。**制約が攻めすぎ**という分かりやすい失敗例。
   → 直すなら:クロックを緩める / パイプライン段追加 / 配置・合成の最適化を強める。
4. **電力も段で変わる** — floorplan 3.08mW(クロック網ゼロ)→ 最終 6.66mW(うちクロック14.2%)。
   クロックツリーを実際に建てて初めてその電力が見える。

クリティカルパスは GCD の 12bit 比較・減算チェーン(`maj3`(多数決ゲート)→`a2111oi`→`o31ai`… の段積み)。
配線後に `rebuffer*` というバッファが挿入されているのは、長配線を区切って駆動するための最適化の痕跡。

## 2. 各段の中身

### 2_floorplan — 土地と電気の用意
- **ダイ/コアの大きさ**と利用率(utilization)を決める。セルをどれだけ詰めるか。
- **I/Oピンの位置**(レイアウト画像の四辺の緑矢印)。
- **電源網 PDN**(画像の水平シアン帯 = met4/met5 の電源ストラップ)。全セルに電源を届ける格子。
- **タップセル / ウェルタップ**を一定間隔で挿入(ラッチアップ対策)。
- この時点ではクロックは「理想配線」のままなので、STAのクロック網遅延=0。

### 3_place — セルを座標に置く
- **global placement**:重なり許容で「だいたい最適」な位置へ(配線長・混雑を最小化)。
- **legalize**:標準セル行(row)にスナップし重なりを解消。
- **detailed placement**:微調整。
- 同時に **配置駆動の最適化**:配線長を見越したバッファ挿入・ゲートサイズ変更(`rebuffer*` セルが生まれる)。
- ここで初めて「どのセルがどれだけ離れているか=配線長」が見えるので slack が動く。

### 4_cts — クロックを等遅延で配る木を建てる
- 全FFのCLKピンに、**できるだけ同時に**クロックが届くようバッファの木(`clkbuf_*`)を構築。
- 指標:**skew**(到達時刻のばらつき、gcdでは 0.01ns と良好)/ **insertion delay**(根から葉までの遅延)。
- ここで初めてクロック配線が「現実」になるので、クロック電力が発生し、slack も動く。

### 5_route — 実際の金属配線を引く
- **global route**:配線混雑(congestion)を見積もり、おおまかな経路を決める。
- **detailed route**:実メタル層に正確に配線。ビア・スペーシングなど設計規則を満たす。
- 終わって **DRC(design rule check)= 0 違反**(gcd)。
- 実配線の RC が確定するので、ここが**タイミング最悪点**になりやすい。

### 6_final — 仕上げと署名(signoff)
- **フィラーセル**で隙間を埋める(ウェル連続性・密度規則のため)。
- **寄生抽出**:実配線の RC を `.spef` に書き出し、それを使って最終 STA。
- **署名STA + 電力レポート**(`6_finish.rpt`):最終的なタイミング・電力の確定値。
- **GDSII 出力**(`6_final.gds`):製造に渡すマスクデータ。`klayout` で開ける。

## 3. PPA という見方

PD の良し悪しは結局 **PPA = Performance(fmax/slack)・Power(mW)・Area(面積/利用率)** のバランス。
3つはトレードオフ:速くしようとバッファを増やせば面積と電力が増える、詰めれば配線が混んで遅くなる、等。
gcd の例だと「Perf は未達(slack赤)/ Power 6.66mW / Area は sky130hd の小設計」で、
**Perf を取りに行くなら制約緩和かRTL側(パイプライン化)が必要**、と読める。

## 4. 実環境メモ(再現用)

- 環境:`~/OpenROAD-flow-scripts`(ORFS)。実行は `./run_flow.sh designs/sky130hd/gcd/config.mk`。
  - i9-13900H は AVX-512 非対応 → OpenROAD をローカル再ビルド済(でないと CTS 段で illegal instruction)。
  - `EQUIVALENCE_CHECK=0` 必須(ローカル yosys/eqy の ABI 不一致回避)。
- 成果物:`flow/results/sky130hd/gcd/base/`(各段の `.odb`/`.def`/`.v`/`.sdc`/`6_final.gds`)。
- レポート:`flow/reports/sky130hd/gcd/base/`(`2_floorplan_final.rpt` … `6_finish.rpt`、レイアウト `.webp`)。
- GDS確認:ホストの `klayout`。

> 関連:設計を「作る」本体は SHISAKU(合成=yosysのみ)。STA/P&R/パワー/GDS はこの ORFS が担当。
